## 0. Purpose & Scope

Define a single, enforceable TypeScript configuration model for the monorepo that matches modern Nx
project-linking guidance and pnpm workspace best practices. Covers root configs, per-project
configs, package.json requirements, and contributor workflow for adding or moving projects.

## 2. Principles

1. **Project References everywhere:** Type-check via `tsc --build` so dependencies are validated
   transitively and incrementally (TS handbook; Nx project linking).
2. **NodeNext for libraries, Bundler only for Next.js apps:** Publishable libs resolve via Node’s
   ESM semantics; bundler resolution stays scoped to Next.
3. **No `paths` or `baseUrl`:** Module boundaries are defined by each package’s `exports` map;
   editors and `tsc` resolve through package.json (Nx, TS guidance).
4. **Split build vs tests:** `tsconfig.lib.json` (composite, emits declarations) vs
   `tsconfig.spec.json` (leaf, test-only types, no emit).
5. **Workspace-first linking:** Internal dependencies use `workspace:*` ranges; pnpm keeps a single
   source of truth and Nx graphs stay accurate.
6. **References are generated, not hand-written:** `nx sync` keeps root and project references
   aligned with the workspace graph.

## 3. Root Configuration

### 3.1 `/tsconfig.base.json` (shared compiler options)

- NodeNext module/resolution, ES2023 target/lib.
- Strict mode + safety flags (`exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`, etc.).
- **Do not** set `baseUrl` or `paths`; removal is enforced.
- `moduleDetection: "force"`, `verbatimModuleSyntax: true`,
  `resolvePackageJsonExports/Imports: true`, `customConditions: ["@repo/source"]`.
- `incremental: true`, `noEmitOnError: true`, `sourceMap: true`, `skipLibCheck: true`.

### 3.2 `/tsconfig.json` (workspace type-check entrypoint)

- Extends `tsconfig.base.json`.
- `files: []`, `references: [...]` only (no `include` globs).
- `compilerOptions.noEmit: true`, `allowImportingTsExtensions: true`, `moduleDetection: "force"`.
- Maintained by `nx sync` so TypeScript graph = Nx project graph.

## 4. Per-Project Configuration

### 4.1 Libraries (Node or React)

File layout inside each package:

- `tsconfig.base.json` → extends `@repo/typescript-config/node-library.json` or
  `react-library.json`; holds compiler options only.
- `tsconfig.json` → **REQUIRED: references BOTH `./tsconfig.lib.json` AND `./tsconfig.spec.json`**;
  `files: []`; no `include`. This ensures tests are included in `tsc --build` (root `nx typecheck`).
- `tsconfig.lib.json` → `composite: true`, `rootDir: src`, `outDir: dist`, `declaration: true`,
  `declarationMap: true`, `sourceMap: true`, `noEmitOnError: true`; `references` list the package’s
  workspace deps.
- `tsconfig.spec.json` → `composite: true`, test-only types (`vitest/globals`, `node`), test-only
  includes; **references the lib config** (`{"path": "./tsconfig.lib.json"}`) so tests importing
  production code satisfy TS build’s requirement that every file belong to a referenced project.

Rationale: referencing the spec config in the solution tsconfig keeps tests in `tsc -b` coverage
while the spec config’s reference to the lib config prevents TS6307 (“file not listed”) without
pulling test-only types into the published surface. **Tests must be type-checked.**

### 4.2 Applications (Next.js)

- `tsconfig.json` generated by Nx/Next; resolution = `Bundler`.
- **REQUIRED: include both `tsconfig.app.json` and `tsconfig.spec.json` as references** in the
  solution `tsconfig.json` for `tsc -b` (root `nx typecheck`) to validate both app and test files.
  Mark `tsconfig.spec.json` as `composite: true` and add a reference to `./tsconfig.app.json` so
  tests can import app code.
- Do not reuse Bundler resolution in libraries. References are maintained via `nx sync`.

## 5. `package.json` Requirements (per package)

- `"type": "module"`.
- `"exports"` map at least `{ ".": { "types": "./dist/index.d.ts", "import": "./dist/index.js" } }`;
  add subpaths as needed.
- Internal deps use `"workspace:^"` (or `"workspace:*"` for dev-only).
- Script: `"check-types": "tsc -b tsconfig.json"` to validate lib + spec configs using incremental
  build artifacts for speed. Optional
  `"check-types:clean": "tsc -b tsconfig.json && tsc -b tsconfig.json --clean"` is provided when a
  zero-artifact pass is required (e.g., before publishing). TypeScript build mode requires emit
  (cannot combine `--build` with `--noEmit` or `--outDir` overrides). (Refs: TypeScript project
  references handbook; https://www.typescriptlang.org/docs/handbook/project-references.html, Nx TS
  workspace guide
  https://nx.dev/technologies/typescript/recipes/switch-to-workspaces-project-references)
- Script: `"build": "tsc -b tsconfig.lib.json"` to build runtime output using project references so
  dependencies are built in topo order. Using `-p tsconfig.lib.json --incremental` can miss
  dependent emits and is not aligned with the Nx TypeScript project-linking guidance. (Refs: Nx TS
  project linking https://nx.dev/concepts/typescript-project-linking)
- `"sideEffects": false` for publishable libraries.
- Keep version fields aligned with workspace policy (semantic versioning; pre-1.0 allowed for
  internal libs).

## 6. pnpm Workspace & Linking

- Use `pnpm` with default `link-workspace-packages: true`.
- Declare all internal dependencies with `workspace:*`/`workspace:^` to avoid duplicate versions and
  to let Nx graph resolve correctly.
- Do not hand-wire symlinks; rely on pnpm workspace protocol.
- When adding a package, run `pnpm install` to refresh lockfile after the `package.json` change.

## 7. Contributor Workflow

1. Generate project with Nx (e.g.,
   `pnpm dlx nx generate @nx/js:library packages/foo --bundler=tsc --unitTestRunner=vitest`).
2. Run `nx sync` to regenerate project references in root `tsconfig.json` and Nx graph.
3. Confirm per-project configs match §4 (lib/spec split, no paths).
4. Update `package.json` with `exports`, `workspace:*` deps, `check-types` script.
5. Run `pnpm post:report:fast` for quick validation; `pnpm post:report` before handoff.

### 7.1 `nx sync` guidance, issues, and remedies

- **What it does:** Regenerates project configuration (including TS references) from workspace
  manifests so Nx, TypeScript, and IDEs share the same graph. (Ref: Nx “switch to workspaces project
  references”).
- **When to run:** After adding, renaming, or moving a project; after changing tags/implicit deps;
  after editing `nx.json`/`project.json` paths.
- **Common issues & fixes:**
  - _New project missing in `tsconfig.json` references_: run `nx sync`; if still absent, ensure the
    project’s `project.json` lives under a workspace pattern and has `root` set correctly.
  - _Stale references after rename/move_: run `nx graph --watch` to confirm layout; delete obsolete
    `tsconfig*.tsbuildinfo` files if TypeScript still points to old paths.
  - _IDE not picking up changes_: restart TS server/IDE after `nx sync`; verify `tsconfig.json` now
    shows the updated `references`.
  - _Bundler/NodeNext mix-ups_: if a library accidentally picks up Bundler resolution, ensure its
    `tsconfig.lib.json` extends the Node preset and re-run `nx sync` so the graph uses the corrected
    config.

## 8. Migration Checklist (existing repos)

- Remove `baseUrl`/`paths` from `/tsconfig.base.json`.
- Rewrite `/tsconfig.json` to references-only; run `nx sync`.
- For each package/app, ensure `tsconfig.json` is references-only; create/normalize
  `tsconfig.lib.json` (composite) and `tsconfig.spec.json` (leaf).
- Add/clean `exports` fields and switch internal deps to `workspace:*`/`workspace:^`.
- Replace any `tsc --noEmit -p tsconfig.json` scripts with `tsc --build tsconfig.json` (optionally
  follow with `tsc -b tsconfig.json --clean` or `pnpm clean` when artifacts must be removed).
- Delete legacy `tsconfig.build.json` stubs once Nx plugins no longer hash them.

## 9. Validation Commands

- `pnpm typecheck` (Nx `check-types`) → `tsc --build` across projects **including test files** (via
  spec config references in each project's `tsconfig.json`).
- `pnpm test` / `pnpm test:coverage` → validates spec configs with test-only types.
- `pnpm lint` → import ordering + module boundary checks complement TS settings.
- `nx graph` → verify project reference topology matches expectations.
