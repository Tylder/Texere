# Texere v1.1 Node Schema Draft

Status: Draft Scope: Semantic memory schema only (LLM-authored nodes)

## Goals

- Reduce type ambiguity for LLMs during node creation.
- Keep ingest complexity separate from semantic authoring complexity.
- Enforce clear, strict combinations without backward compatibility.

## Non-Goals

- No API versioning (tool names stay the same).
- No backward compatibility for legacy node types.
- No attempt to represent every nuance as a top-level node type.

## Core Model

Each semantic node has:

- `type` (core type, required)
- `title` (required)
- `content` (required)
- `role` (required, constrained by `type`)
- `source` (optional, default: `internal`)
- `status` (optional, default: `active`)
- `scope` (optional, default: `project`)
- `tags` (optional)
- `importance` (optional, 0-1)
- `confidence` (optional, 0-1)
- `anchor_to` (optional)

## Core Types (`type`)

- `knowledge`
- `issue`
- `action`
- `artifact`
- `context`
- `meta`

## Facets

### Role (`role`)

Allowed role values:

- `decision`
- `requirement`
- `constraint`
- `principle`
- `finding`
- `example`
- `research`
- `problem`
- `error`
- `task`
- `solution`
- `fix`
- `workflow`
- `command`
- `code_pattern`
- `technology`
- `project`
- `file_context`
- `conversation`
- `system`

### Source (`source`)

- `internal`
- `external`

### Status (`status`)

- `proposed`
- `active`
- `deprecated`
- `invalidated`

### Scope (`scope`)

- `project`
- `module`
- `file`
- `session`

## Strict Type-Role Constraint Matrix

Valid combinations:

- `knowledge` -> `decision|requirement|constraint|principle|finding|research`
- `issue` -> `problem|error`
- `action` -> `task|solution|fix|workflow|command`
- `artifact` -> `example|code_pattern|technology|project|file_context`
- `context` -> `conversation`
- `meta` -> `system`

All other `type` + `role` combinations are invalid.

## Validation Rules

- Reject unknown `type`.
- Reject unknown `role`, `source`, `status`, `scope`.
- Reject invalid `type` + `role` pair.
- Reject empty `title` or `content`.
- Reject `importance`/`confidence` outside 0-1.

## Ingest vs Semantic Separation

Two concerns remain distinct:

- Automated ingest (code-driven): may use richer internal classifications.
- Semantic nodes (LLM-authored): must conform to this strict schema.

Ingest output should be projected into semantic nodes via deterministic mapping to `type` + `role`.

## Tool Contract Impact (No Versioning)

`texere_store_node` remains the same tool name, with updated input semantics:

- `type` now uses the v1.1 core type set.
- `role` becomes required.
- Legacy types are not accepted.

## Open Questions to Resolve Before Implementation

- Should `source` default to `internal` for all writes, including ingest?
- Should `status=invalidated` remain edge-driven only (`DEPRECATED_BY`) or be directly writable?
- Do we want `meta/system` user-writable, or reserved for internal workflows only?
