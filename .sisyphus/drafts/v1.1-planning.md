# Draft: Texere v1.1 Planning

## Requirements (confirmed)

- **Scope**: v1.1 refinement, not v2. Schema + behavior changes, no new API versions.
- **No backward compatibility**: hard cutover from current 17-type enum. Old values become invalid
  immediately.
- **No API versioning**: tool names stay the same (`texere_store_node`, `texere_create_edge`, etc.).
- **Two separate concerns**: automated code ingest (can be complex) vs semantic LLM-authored nodes
  (must be simple).

## Decisions Made

### 1. Node Type Simplification (Schema)

**Decision**: Replace 17 node types with 6 core types + required `role` facet.

**Core types**: `knowledge`, `issue`, `action`, `artifact`, `context`, `meta`

**Facets**:

- `role` (required, constrained by type)
- `source` (optional, default: `internal`) — values: `internal`, `external`
- `status` (optional, default: `active`) — values: `proposed`, `active`, `deprecated`, `invalidated`
- `scope` (optional, default: `project`) — values: `project`, `module`, `file`, `session`

**Strict type-role constraint matrix**:

- `knowledge` -> `decision|requirement|constraint|principle|finding|research`
- `issue` -> `problem|error`
- `action` -> `task|solution|fix|workflow|command`
- `artifact` -> `example|code_pattern|technology|project|file_context`
- `context` -> `conversation`
- `meta` -> `system`

Invalid combinations are rejected.

**Rationale**: 6-way type decision is obvious for agents; semantic precision moves to facets. Ingest
layer can use richer internal classifications and project into this schema.

**Schema draft**: `docs/v1.1-node-schema-draft.md`

### 2. Batch Create Semantics

**Decision**: All create methods (`texere_store_node`, `texere_create_edge`) accept single object OR
array.

- Input is normalized internally to a common batch pipeline.
- Single object input works exactly as before (no regression).
- Array input processes all items in one call.
- **Always atomic**: any failure aborts entire operation, zero writes committed.
- No `atomic` flag — atomicity is not optional, it's the only mode.
- Pre-validate full payload before opening transaction (fail fast on schema issues).
- Return per-item outcomes on success (`index`, `id`).
- Return deterministic failure details on abort (`failed_index`, `error_code`, `message`) with no
  partial success payload.
- Agents know retries are safe; no reconciliation logic needed.

**Rationale**: Zero regression for agent one-at-a-time flows; immediate bulk ergonomics for
modeling/import workflows. All-or-nothing is simpler to reason about than mixed modes.

### 3. Search Result Explainability

**Decision**: Expose retrieval metadata in search/about results.

- Return `rank` (raw BM25 score, currently computed then dropped at `search.ts:77`).
- Return normalized `match_quality` (0-1).
- Return `match_fields` array (which fields matched: `title`, `content`, `tags`).

**Rationale**: LLM trust drops when relevance is opaque. Agents can't assess why a result ranked
high without this. External research (mem0, LangChain, Elasticsearch Labs) all converge on score
transparency.

### 4. Relationship Summary in Search Results

**Decision**: Add lightweight relationship counts to `texere_search` and `texere_about` payloads.

- Per result:
  `relationships: { outgoing: { SOLVES: 1, REQUIRES: 2 }, incoming: { MOTIVATED_BY: 1 } }`
- Counts by edge type, not full edge objects.

**Rationale**: Agents currently need extra traversal calls to understand graph context. Single-call
relationship awareness reduces round-trips.

### 5. Minimal Response Mode

**Decision**: Add `minimal: true` option to write tools (`texere_store_node`, `texere_create_edge`).

- Default: `false` (current verbose behavior, no regression).
- When `true`: return only `{ id }` (or `[{ id }, ...]` for batch).
- ~90% token reduction per write response.

**Rationale**: Current full payload (~180 tokens) is wasteful when agent only needs the ID for the
next operation. Especially impactful for batch workflows.

### 6. Duplicate Warning on Create

**Decision**: Add non-blocking duplicate detection on `texere_store_node`.

- Preflight similarity check before insert (fuzzy title matching).
- Returns warning with similar node IDs if duplicates detected.
- Does NOT block creation — informational only.
- Agent decides whether to proceed or use existing node.

**Rationale**: Prevents accidental duplicates that pollute graph quality. Non-blocking preserves
fast write path.

## Research Findings

### From Internal Code Audit

- BM25 rank is computed (`search.ts:67`) but stripped before return (`search.ts:77`) — easy fix.
- FTS5 boolean/phrase capability is overstated in docs; sanitizer strips quotes and escapes
  operators (`sanitize.ts:8-14`).
- Tag filtering is AND-only; no OR logic for tags (`search.ts:44-52`).
- Type filtering is single-type only; no multi-type OR queries (`search.ts:20-22`).
- Batch ops were intentionally deferred earlier for agent-by-agent writes
  (`memory-graph-audit.md:142`), now reconsidered for modeling workflows.
- `about` tool is confirmed as a strong differentiator (search + traverse in one call).

### From External Research

- mem0: multi-stage scoring (vector similarity + rerank score), hash-based dedup.
- LangChain: search type transparency, score thresholds for quality filtering.
- Elasticsearch Labs: bounded iterative expansion, graph pruning, cardinality limits.
- Academic research: ranking transparency builds agent trust; temporal decay helps relevance;
  deprecation context > silent exclusion.
- All sources confirm: "smart agent, dumb store" is compatible with richer retrieval metadata.

### From Critique Docs (Calibrated)

- Node type confusion: **valid** — addressed by simplification to 6 types + facets.
- Missing `principle`/`example` types: **valid concern, different solution** — now handled as roles
  under `knowledge`/`artifact`.
- Missing `EXEMPLIFIES`/`SUPPORTS` edges: **still open** — not yet decided for v1.1.
- Batch ops pain: **valid** — addressed by array-accepting create methods.
- Verbose responses: **valid** — addressed by minimal response mode.
- Duplicate detection: **valid** — addressed by non-blocking warning.
- FTS5 boolean power claims: **overstated** — sanitizer limits actual capability.

## Open Questions

- Should edge types also be simplified/expanded for v1.1? (Missing `EXEMPLIFIES`, `SUPPORTS` from
  critique.)
- Should FTS5 sanitizer be relaxed to allow actual phrase search and boolean operators?
- Should tag filtering support OR logic in addition to AND?
- Should search support multi-type filtering (`type: [knowledge, action]`)?
- Exact payload shape for batch failure responses?
- Max batch size guardrail value?
