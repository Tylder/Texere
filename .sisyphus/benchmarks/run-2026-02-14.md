## Workflow Friction

### Search-before-create

What felt clunky:
- Requirement is explicit, but high-volume ingestion makes per-node search repetitive.
- For 58 nodes, search overhead dominates ingestion mechanics even when graph is empty.

Why it is friction:
- High ceremony for low-signal operations in cold-start graphs.
- Adds cognitive and token overhead before value-adding modeling work.

Actionable improvement:
- Add `store_nodes_with_dedup` that internally performs search + duplicate detection + creation in one atomic call.

### Batch operations

What felt clunky:
- Node batching is good, but relationship creation still required manual ID bookkeeping.
- Opaque IDs force error-prone mapping between conceptual plan and final edge construction.

Actionable improvement:
- Support temporary client keys in `store_nodes` and allow `create_edges` to reference those keys directly in same transaction.

### Provenance modeling

What felt clunky:
- Distinguishing `ABOUT` vs `BASED_ON` for doc provenance is ambiguous.

Actionable improvement:
- Add explicit guidance matrix (when to use `ABOUT` vs `BASED_ON`) with ingestion-oriented examples.

## Type System Findings

### Ambiguities encountered

1. `principle` vs `decision`
- Example: `Texere should remain JSON-first for LLM tool consumers` (`IgUH8nl-v4BFv3x6bqQrk`).
- Ambiguity: normative guidance (`principle`) vs explicit architectural choice (`decision`).

2. `constraint` vs `pitfall`
- Example: `sql.js lacks WAL and default FTS5 support for MCP usage` (`meqcQ8JfkF3LWqpU5Dxrw`).
- Ambiguity: hard capability limit (`constraint`) but also operational hazard (`pitfall`).

3. `SUPPORTS` vs `ABOUT` vs `BASED_ON`
- Example cluster around search UX nodes from `search-comparison-llm-ux.md`.
- Ambiguity: provenance, topical relation, and argumentative support are all distinct semantics but close in practice.

### Missing types

Potential node roles that would reduce ambiguity:
- `hypothesis` (currently forced into `finding`/`research`).
- `tradeoff` (currently split across content text and generic role choices).

Potential edge types that would reduce ambiguity:
- `EVIDENCE_FOR`, `TRADEOFF_WITH`, `DERIVED_FROM`.

### Well-designed aspects

- Enum-constrained node/edge types prevented schema drift.
- Separation of `issue` and `knowledge` produced cleaner problem-solution modeling.
- `RESOLVES` is especially useful for linking requirements/solutions to concrete problems.

## SKILL.md Documentation Gaps

1. Search-before-create at scale
- Missing practical guidance for large ingestion runs (50-200 planned nodes).
- No recommended pattern for balancing strict dedup workflow with batch throughput.

2. Provenance edge semantics
- `ABOUT` vs `BASED_ON` choice lacks concrete examples for doc ingestion and benchmark evidence extraction.

3. Ambiguity handling
- No explicit decision rubric for frequent boundary cases (`decision` vs `principle`, `constraint` vs `pitfall`).

4. Batch dedup workflow
- Missing end-to-end recipe combining pre-search, dedup, node creation, and edge creation with minimal manual mapping.

## Tool Friction

- `texere_stats` lacks role-level aggregate counts; role distribution had to be tracked manually.
- No dry-run "graph plan validation" for large node+edge batches before write.
- Search response did not provide lightweight duplicate-confidence summary tailored for ingestion workflows.

## Recommendations

1. Add an atomic ingestion API
- `ingest_nodes({ candidates, dedup: true, provenance_source })` to collapse repetitive search/create steps.

2. Improve provenance semantics
- Add edge type `DERIVED_FROM` (or strict guidance to disambiguate `ABOUT` vs `BASED_ON`).

3. Expand analytics output
- Extend `texere_stats` with `byRole` and optional time-scoped run summaries.

4. Add ingestion planner support
- Provide dry-run mode that returns proposed nodes/edges, dedup collisions, and suggested edge mappings before committing.

5. Add ambiguity playbook in SKILL.md
- Include common boundary cases and canonical examples from real ingestion scenarios.
