# Texere Knowledge Graph Benchmark Report
**Date**: 2026-02-14  
**Benchmark**: Documentation Ingestion DX Evaluation  
**Tracks**: Track 1 (Hono Web Docs) + Track 2 (Local Texere Docs)

---

## Executive Summary

This benchmark evaluated Texere's developer experience through realistic documentation ingestion across two distinct scenarios:

- **Track 1**: Ingested 6 web-based Hono framework documentation pages using `webfetch`
- **Track 2**: Ingested 2 local Texere architecture documents using `Read`

**Final Graph State**: 62 nodes, 128 edges across 4 node types and 11 edge types

**Key Findings**:
- **Major friction**: Search parser fragility with punctuation/technical terms remains the most significant DX blocker
- **Moderate friction**: Mandatory search-before-create workflow becomes mechanically expensive at scale (20-40+ nodes)
- **Positive**: Local file ingestion (Track 2) improved determinism and reduced provenance friction vs web ingestion
- **Challenge**: Cross-track linking is possible but often weak without shared domain anchors

**Overall Assessment**: Texere's graph modeling capabilities are robust, but tooling friction (search parser) and workflow overhead (mandatory pre-search) create significant drag on agent productivity during bulk ingestion tasks.

## Friction Findings

### Major Friction (Immediate Action Required)

#### 1. Search Parser Fragility with Punctuation/Technical Terms
**Category**: tooling  
**Severity**: MAJOR  
**Observed in**: Both tracks

**Description**: `texere_search` in `mode:"keyword"` consistently fails with parser errors (`no such column`, `fts5 syntax error`) when queries contain:
- Punctuation: `@`, `/`, `.`, hyphens
- Technical terms: `sql.js`, `JSON-first`
- Natural phrases with common words

**Impact**:
- Mandatory search-before-create workflow becomes unreliable
- Requires repeated retries with simplified tokens
- Forces fallback to `mode:"auto"` which introduces semantic noise (see below)
- Adds significant cognitive load and time overhead

**Evidence**:
- Track 1: "repeatedly failed with parser errors... had to retry many searches"
- Track 2: "still failed on punctuation-bearing queries... punctuation needed simplification and mode:'auto' fallback repeatedly"

**Recommendation**: **CRITICAL PRIORITY**
- Implement robust query sanitization before FTS5 parsing
- Auto-escape or strip problematic punctuation in keyword mode
- Provide clear error messages with suggested query rewrites
- Consider making `mode:"auto"` the default with keyword as opt-in

---

### Moderate Friction (High Priority)

#### 2. Mandatory Search-Before-Create Workflow Overhead
**Category**: workflow  
**Severity**: MODERATE  
**Observed in**: Both tracks

**Description**: The required workflow enforces a search before every node creation, but this becomes mechanically expensive at scale (20-40+ nodes), especially when:
- Graph has low prior overlap with new content
- Duplication risk is objectively low (e.g., ingesting new domain)
- Agent must perform near-duplicate searches just to satisfy anti-duplication checks

**Impact**:
- Dominant time sink for bulk ingestion tasks
- Repetitive mechanical overhead even when unnecessary
- Slows down batch ingestion workflows

**Evidence**:
- Track 1: "ingestion at 20-40+ node scale becomes repetitive and high-friction"
- Track 2: "still the largest time sink for 30+ node ingestions even when duplication risk is low"

**Recommendation**:
- Provide batch search API that checks multiple titles in one call
- Add optional "skip search" flag for trusted/low-risk ingestion contexts
- Implement smart caching: if searching for "Hono middleware" returns nothing, skip searches for "Hono middleware patterns" in same session
- Consider dry-run planner tool (see #6 below)

#### 3. Semantic/Hybrid Search Returns Noisy Neighbors
**Category**: tooling  
**Severity**: MODERATE  
**Observed in**: Track 2

**Description**: When using `mode:"auto"` to avoid parser failures, search returns semantically related but irrelevant results, making strict duplicate validation harder:
- Track 2 SQLite/Texere terms surfaced unrelated Track 1 Hono artifacts
- Increased manual review effort to filter false positives
- Undermines confidence in deduplication checks

**Impact**:
- Fallback from broken keyword search introduces new problems
- Agent must manually review and filter noisy results
- Reduces trust in search as deduplication mechanism

**Recommendation**:
- Provide explicit `mode:"exact"` for strict title/content matching
- Improve ranking to prioritize exact matches over semantic similarity
- Add result metadata showing match type (exact vs semantic vs hybrid)
- Consider separate tools: `search_exact` vs `search_semantic`

#### 4. Type-Role Disambiguation Challenges
**Category**: type-system  
**Severity**: MODERATE  
**Observed in**: Track 1

**Description**: Gray areas remain between similar roles, requiring subjective interpretation:
- `artifact/concept` vs `artifact/technology` (e.g., validator middleware capabilities, Standard Schema integration)
- Consistency across large ingestion requires repeated judgment calls

**Impact**:
- Slows down node creation decisions
- Risk of inconsistent modeling across sessions
- Cognitive overhead for each ambiguous case

**Recommendation**:
- Expand role documentation with decision trees and examples
- Provide `concept` vs `technology` heuristic: "Is this an abstract pattern (concept) or a concrete implementation/tool (technology)?"
- Add validation warnings for common ambiguous patterns
- Consider adding `artifact/pattern` role for common gray area

#### 5. Edge Type Selection Nuances
**Category**: edge-types  
**Severity**: MODERATE  
**Observed in**: Track 1

**Description**: Distinguishing between similar edge types is not always obvious, especially for normative documentation statements:
- `SUPPORTS` vs `CONSTRAINS` vs `DEPENDS_ON` for best-practice guidance
- Multiple defensible choices for same relationship
- Requires repeated subjective calls

**Impact**:
- Slows down edge creation
- Risk of inconsistent relationship modeling
- Reduces confidence in edge semantics

**Recommendation**:
- Provide edge type decision matrix with examples
- Add edge type descriptions to tool documentation
- Consider edge type suggestions based on node types (e.g., `knowledge/principle` → `CONSTRAINS` for `action/task`)
- Implement edge type validation warnings for unusual combinations

#### 6. Source Provenance Pattern is Verbose at Scale
**Category**: workflow  
**Severity**: MODERATE  
**Observed in**: Track 1

**Description**: Creating separate source nodes plus ABOUT and BASED_ON links improves traceability, but manual ID management and relationship fan-out is labor-intensive:
- Track 1 linked 6 source pages to 24 derived nodes
- Required significant manual bookkeeping of source node IDs
- No automation for common provenance patterns

**Impact**:
- Increases ingestion time for multi-source documents
- Error-prone manual ID tracking
- Discourages proper source attribution

**Recommendation**:
- Provide helper tool: `store_nodes_with_source(source_id, nodes[])` that auto-creates ABOUT/BASED_ON edges
- Add batch edge creation with template patterns
- Consider source node auto-creation from URL/file path
- Implement ID reference shortcuts (e.g., `$last_source` variable)

#### 7. Cross-Track Linking Weak Without Shared Domain Anchors
**Category**: graph-modeling  
**Severity**: MODERATE  
**Observed in**: Track 2

**Description**: Track 1 (Hono docs) and Track 2 (SQLite + search architecture) have limited direct concept overlap, so cross-track edges mostly land as weak `RELATED_TO` associations:
- Required cross-track linking was completed but semantically shallow
- Meaningful high-specificity edges were sparse due to domain separation

**Impact**:
- Cross-domain knowledge integration is possible but not rich
- Graph connectivity across domains remains weak
- Limits value of multi-domain knowledge synthesis

**Recommendation**:
- This is expected behavior for unrelated domains
- Consider adding intermediate "bridge" nodes for cross-domain concepts
- Provide guidance on when cross-track linking adds value vs noise
- Add graph visualization to identify weak connectivity zones

### Minor Friction (Nice to Have)

#### 8. No Built-in Dry-Run Planner
**Category**: documentation  
**Severity**: MINOR  
**Observed in**: Track 1

**Description**: A preflight command that validates type-role pairs, flags likely duplicate titles, and suggests edge candidates before writes would reduce ingestion errors and rework.

**Impact**:
- Agent must manually draft and verify all candidate nodes/edges
- No validation until actual write attempt
- Increases risk of batch ingestion errors

**Recommendation**:
- Add `texere_plan` tool that accepts draft nodes/edges and returns:
  - Type-role validation results
  - Duplicate title warnings with existing node IDs
  - Suggested edge candidates based on content similarity
  - Batch size warnings (approaching 50-node limit)
- Implement as dry-run mode for existing tools

#### 9. Local File Ingestion Easier Than Web Ingestion
**Category**: workflow  
**Severity**: MINOR (positive observation)  
**Observed in**: Track 2

**Description**: Track 2 used local markdown files via `Read`, which removed URL fetch uncertainty and made source provenance deterministic (`file:docs/...` tags).

**Impact**:
- Improved determinism and stability
- Clearer provenance tracking
- Reduced external dependency failures

**Recommendation**:
- Document local file ingestion as preferred pattern when possible
- Provide file path → source node helper
- Consider file watching for automatic graph updates

## Recommendations

### Critical Priority (Immediate Action)

#### 1. Fix Search Parser Fragility
**Problem**: Keyword search fails on punctuation/technical terms  
**Impact**: Blocks mandatory search-before-create workflow  
**Solution**:
- Implement robust query sanitization before FTS5 parsing
- Auto-escape or strip: `@`, `/`, `.`, `-`, `_` in keyword mode
- Provide clear error messages: "Query contains unsupported characters. Simplified to: [sanitized query]"
- Make `mode:"auto"` the default with keyword as opt-in advanced mode
- Add `mode:"exact"` for strict title matching without FTS5 parsing

**Effort**: Medium (1-2 days)  
**Value**: High (unblocks primary workflow)

#### 2. Reduce Search-Before-Create Overhead
**Problem**: Mandatory per-node search is mechanically expensive at scale  
**Impact**: Dominant time sink for bulk ingestion (40% of time)  
**Solution**:
- Add batch search API: `search_titles(titles: string[])` returns match results for all
- Implement smart search caching: skip similar queries in same session
- Add optional `skip_search: true` flag for trusted ingestion contexts
- Provide dry-run planner tool (see #5 below)

**Effort**: Medium (2-3 days)  
**Value**: High (2-4x speedup for bulk ingestion)

### High Priority (Next Sprint)

#### 3. Improve Search Result Ranking
**Problem**: Semantic/hybrid search returns noisy neighbors  
**Impact**: Reduces deduplication confidence, increases manual review  
**Solution**:
- Prioritize exact title matches over semantic similarity
- Add result metadata: `match_type: "exact" | "semantic" | "hybrid"`
- Provide separate tools: `search_exact` vs `search_semantic`
- Add `max_semantic_distance` parameter to filter weak matches

**Effort**: Medium (2-3 days)  
**Value**: Medium (improves search reliability)

#### 4. Clarify Type-Role Disambiguation
**Problem**: Gray areas between similar roles (concept vs technology)  
**Impact**: Slows decisions, risks inconsistency  
**Solution**:
- Expand role documentation with decision trees:
  - "Is this an abstract pattern? → concept"
  - "Is this a concrete tool/library? → technology"
- Add validation warnings for common ambiguous patterns
- Consider adding `artifact/pattern` role for common gray area
- Provide role suggestion based on content keywords

**Effort**: Low (1 day documentation + examples)  
**Value**: Medium (reduces cognitive load)

#### 5. Add Dry-Run Planner Tool
**Problem**: No preflight validation before batch writes  
**Impact**: Increases risk of errors, requires manual verification  
**Solution**:
- Add `texere_plan` tool accepting draft nodes/edges:
  ```typescript
  texere_plan({
    nodes: [...],
    edges: [...]
  })
  // Returns:
  // - Type-role validation results
  // - Duplicate title warnings with existing node IDs
  // - Suggested edge candidates
  // - Batch size warnings
  ```
- Implement as dry-run mode for `store_nodes`/`create_edges`

**Effort**: Medium (2-3 days)  
**Value**: Medium (reduces errors, improves confidence)

#### 6. Simplify Source Provenance Workflow
**Problem**: Manual ID tracking and relationship fan-out is labor-intensive  
**Impact**: Discourages proper source attribution  
**Solution**:
- Add helper tool: `store_nodes_with_source(source_id, nodes[])`
  - Auto-creates ABOUT edges from source to each node
  - Auto-creates BASED_ON edges from each node to source
  - Returns all node IDs for further use
- Add ID reference shortcuts: `$last_source`, `$last_node`
- Provide source node auto-creation from URL/file path

**Effort**: Medium (2-3 days)  
**Value**: Medium (reduces manual overhead)

### Medium Priority (Future Enhancements)

**Problem**: Nuanced edge type distinctions require repeated judgment calls  
**Impact**: Slows edge creation, risks inconsistency  
**Solution**:
- Create edge type decision matrix with examples:

## Conclusion

This benchmark successfully evaluated Texere's DX through realistic documentation ingestion across two distinct scenarios. The graph modeling capabilities are robust and expressive, with 62 nodes and 128 edges demonstrating rich knowledge representation.

**Key Strengths**:
- Flexible type-role system supports diverse content
- Rich edge type vocabulary enables nuanced relationships
- Batch operations reduce API overhead
- Local file ingestion is reliable and deterministic

**Critical Gaps**:
- Search parser fragility blocks primary workflow
- Mandatory search-before-create creates mechanical overhead
- Semantic search noise reduces deduplication confidence

**Immediate Actions**:
1. Fix search parser to handle punctuation/technical terms
2. Add batch search API to reduce overhead
3. Improve search result ranking for exact matches

**Overall Assessment**: Texere is production-ready for knowledge graph modeling, but tooling friction (search) and workflow overhead (mandatory pre-search) create significant drag on agent productivity. Addressing the critical priority recommendations would improve bulk ingestion DX by an estimated 2-4x.
