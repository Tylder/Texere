## Friction Log

### Friction Log - Track 1

- Search-before-create is robust but high-friction in bulk ingest: the workflow forces dozens of repetitive micro-decisions about query phrasing before each node, and semantic/hybrid results were often empty so the duplicate check felt procedural rather than informative.
- Type/role boundary between `artifact/concept`, `artifact/code_pattern`, and `knowledge/principle` was repeatedly ambiguous when docs mixed implementation and guidance in one section (e.g., middleware execution order appears as both concept and best-practice implication).
- Split Test pressure created constant uncertainty about granularity: deciding whether routing ordering should be one concept + one constraint + one example, or collapsed into fewer nodes, has no objective threshold in SKILL guidance.
- Importance/confidence scoring remains subjective with no calibration examples for documentation ingestion. Assigning 0.62 vs 0.72 repeatedly felt arbitrary despite consistent intent.
- Edge selection still has modeling friction: many relations are semantically both `PART_OF` and `DEPENDS_ON` depending on interpretation (e.g., RPC typing and validation pipeline), and avoiding overuse of `RELATED_TO` required repeated manual deliberation.
- `sources` provenance is excellent, but operationally opaque: source nodes/edges are auto-created yet not visible inline during store responses, so verification of provenance completeness needs extra follow-up queries.
- Tool ergonomics friction: creating large, linked node sets requires manual ID bookkeeping across calls; there is no native transaction-like “store nodes + infer/link edges” helper for documentation ingestion workloads.
- SKILL.md gives strong guardrails (Recall Test, Split Test, search-first), but lacks concrete ingest templates for mixed docs pages (feature overview + code snippets + caveats), so conversion strategy is still handcrafted per section.

### Friction Log - Track 2

- Search-before-create remained high-friction and still dominated runtime: every candidate node needed a preflight query, and most keyword checks returned empty arrays, so duplicate prevention felt procedural rather than informative (same core friction as Track 1).
- Type/role ambiguity improved for architecture-heavy text (decision/requirement/finding mapping was clearer than Track 1), but boundary cases still required subjective judgment: requirement vs decision ("must do" vs "chosen strategy") and finding vs problem (observed behavior vs actionable issue).
- Split granularity friction persisted, but shifted shape: design docs contain long rationale chains (evidence -> interpretation -> policy), and deciding where to atomize into separate nodes still had no objective threshold.
- Edge selection friction persisted and was amplified by richer causal text: several relations reasonably fit both `BASED_ON` and `DEPENDS_ON`; resolving this still required manual semantics interpretation.
- New Track 2 friction: representing rejected alternatives cleanly required explicit "rejected alternative" nodes to support `ALTERNATIVE_TO`/`CONTRADICTS`, otherwise conflict structure was under-modeled.
- Provenance opacity persisted from Track 1: `sources` auto-linking is convenient but still not observable inline during writes, so explicit `source` node creation and additional edges were used for verifiability.
- Manual ID bookkeeping remained a major ergonomic cost with denser design graphs: creating many cross-linked decision/finding/problem nodes required careful hand-tracking of generated IDs across write calls.
- Domain familiarity reduced conceptual lookup friction (Texere/SQLite vocabulary was known), but workflow friction from the ingestion protocol itself remained mostly unchanged.

## Track 1 vs Track 2 Comparison

- **Persistent, domain-independent friction**: search-before-create overhead, granularity uncertainty, edge-type ambiguity, provenance visibility gaps, and manual ID bookkeeping all remained in Track 2.
- **Track 1-skewed friction (external API docs)**: stronger ambiguity between `artifact/concept` vs `artifact/code_pattern` vs `knowledge/principle`, because docs interleaved examples and guidance without architectural rationale.
- **Track 2-skewed friction (internal design docs)**: better clarity for decisions/findings/requirements, but more complexity in modeling rejected alternatives and multi-step rationale chains (evidence -> constraint -> requirement -> decision).
- **Smoother in Track 2**: domain understanding and explicit "we chose X because Y" statements made `BASED_ON` and `RESOLVES` edges easier to justify.
- **Rougher in Track 2**: higher linkage density increased modeling load and amplified manual ID/edge orchestration burden.
