**Purpose**: Comprehensive SQLite performance research for knowledge graph workloads

- Constraint: WAL mode mandatory (12-33% boost + concurrency)

## Friction Points & DX Issues

### 1. **Importance/Confidence Assessment** (Severity: Medium)

**Issue**: Required fields `importance` (0-1) and `confidence` (0-1) have no clear guidance for assessment.

**Example**: When creating a node for "Hono Web Framework", is importance 0.9 because it's the main subject? Or 0.7 because it's just documentation? Is confidence 1.0 because it's from official docs? Or 0.95 because docs might be outdated?

**Impact**: Forces arbitrary guesses, inconsistent scoring across nodes. Hard to query by importance later if scale is subjective.

**SKILL.md Guidance**: None. Fields are documented as required but no guidance on how to assess them.

**Recommendation**: 
- Add scoring guidelines to SKILL.md (e.g., importance 0.9+ = core architecture decisions, 0.7-0.8 = important patterns, 0.5-0.6 = nice-to-know details)
- OR make these optional with defaults (importance: 0.5, confidence: 0.8)
- OR add validation warnings if values are suspiciously uniform

---

### 2. **temp_id Scope Limitation** (Severity: High)

**Issue**: `temp_id` only works within a single tool call. Cannot reference temp_ids from previous calls.

**Example**:
```typescript
// Call 1: Create validation pattern
texere_store_artifact({
  nodes: [{ temp_id: "validation_pattern", ... }]
})
// Returns: { id: "Z0mO59QBKKWhtelL5n_Qa", temp_id: "validation_pattern" }

// Call 2: Try to link to it
texere_store_knowledge({
  nodes: [{ temp_id: "validation_pitfall", ... }],
  edges: [{ source_id: "validation_pitfall", target_id: "validation_pattern", type: "RELATED_TO" }]
})
// ERROR: target_id "validation_pattern" not found
```

**Workaround**: Save returned real IDs, use them in subsequent calls. Requires manual bookkeeping.

**Impact**: Breaks atomic workflow for multi-type node creation. Cannot create cross-type relationships in one pass.

**SKILL.md Guidance**: Documents "temp_ids are call-scoped only" but easy to miss. Error message doesn't explain the root cause clearly.

**Recommendation**:
- Allow temp_id references to nodes created in the same session (last N calls)
- OR provide a `texere_store_multi` tool that accepts heterogeneous node types + edges
- OR improve error message: "temp_id 'X' not found. Note: temp_ids only work within a single call. Use the real ID returned from the previous store operation."

---

### 3. **Source Node Auto-Creation Visibility** (Severity: Low)

**Issue**: When using `sources` field, source nodes are auto-created with BASED_ON edges. But there's no easy way to verify what was created or retrieve their IDs.

**Example**:
```typescript
texere_store_knowledge({
  nodes: [{ 
    title: "...", 
    content: "...",
    sources: ["https://hono.dev/docs/", "https://example.com/api"]
  }]
})
// What source node IDs were created? How do I reference them later?
```

**Impact**: Can't create additional edges to auto-created source nodes. Feels like "magic" happening behind the scenes.

**SKILL.md Guidance**: Documents auto-creation but doesn't explain how to access created source nodes.

**Recommendation**:
- Return created source node IDs in response: `{ nodes: [...], sources: [{ url: "...", id: "..." }] }`
- OR add `include_sources: true` param to get_node() to return linked source nodes

---

### 4. **Repetitive URLs in sources Field** (Severity: Low)

**Issue**: Must type full URLs repeatedly when ingesting multiple nodes from same document.

**Example**:
```typescript
// Node 1
sources: ["https://hono.dev/docs/getting-started/basic"]

// Node 2  
sources: ["https://hono.dev/docs/getting-started/basic"]

// Node 3
sources: ["https://hono.dev/docs/getting-started/basic"]
```

**Impact**: Typo risk, verbose, tedious for bulk ingestion.

**Recommendation**:
- Allow URL shorthand: `sources: ["@hono/basic"]` with URL registry/config
- OR add `base_source` param at batch level
- OR tolerate duplicate source URLs (dedupe by content in auto-creation)

---

### 5. **Edge Type Selection Ambiguity** (Severity: Medium)

**Issue**: Unclear which edge type to use for certain relationships. Decision tree in SKILL.md helps but some cases are ambiguous.

**Examples**:
- "Validation pattern" → "Hono framework": Is this PART_OF (component) or RELATED_TO (general association)?
- "Performance finding" → "RegExpRouter": Is this EXAMPLE_OF (demonstrates concept) or RELATED_TO?
- "WAL constraint" → "better-sqlite3 decision": Is this DEPENDS_ON (requires) or RELATED_TO?

**Impact**: Inconsistent edge usage, harder to query by semantic relationships later.

**SKILL.md Guidance**: Has decision tree but doesn't cover all cases. "Use specific type first, RELATED_TO as last resort" is vague.

**Recommendation**:
- Add more examples to decision tree
- OR make RELATED_TO accept optional `relationship_hint` field for context
- OR add validation warnings when RELATED_TO is overused (> 50% of edges)

---

### 6. **Search Before Creating Workflow** (Severity: Medium)

**Issue**: SKILL.md mandates "ALWAYS search before creating nodes" but provides no tooling to enforce or streamline this.

**Example**:
```typescript
// Step 1: Search manually
texere_search({ query: "Hono framework" })

// Step 2: Review results manually (empty)

// Step 3: Create node
texere_store_artifact({ nodes: [...] })
```

**Impact**: Easy to forget search step, manual review is tedious, no automation.

**Recommendation**:
- Add `auto_search: true` param to store tools - automatically searches before creating
- OR add validation mode that warns if similar nodes exist
- OR provide `texere_upsert` that searches + creates/updates atomically

---

### 7. **Minimal Response Default** (Severity: Low)

**Issue**: `minimal: true` is the default for store tools. Only returns `{ id }`, must set `minimal: false` to get full node back.

**Impact**: When ingesting, often want to see what was created (verify tags, check auto-created fields). Defaults to minimal = extra param needed.

**Observation**: This is documented as intentional for bulk operations. But for benchmarking/manual ingestion, verbose mode would be better default.

**Recommendation**:
- Consider `minimal: false` as default (breaking change)
- OR add environment/config flag for default response verbosity

---

### 8. **Agent Usage Reminders** (Severity: Very Low)

**Issue**: After using webfetch directly, got repeated "Agent Usage Reminder" messages suggesting to use explore/librarian agents instead.

**Impact**: Distracting during benchmarking. Messages are helpful for production use but not appropriate for this evaluation task.

**Observation**: These are system messages, not Texere-specific. But worth noting as UX friction.

- `artifact/code_pattern` (reusable pattern)

**Resolution**: Used `code_pattern` because it's meant to be copied/adapted, not just demonstrated once.

**Recommendation**: Clarify in SKILL.md: code_pattern = reusable template, example = specific instance demonstrating a concept.

## Workflow Steps That Felt Clunky

### 1. Multi-Type Ingestion Requires Multiple Calls

**Scenario**: Ingesting Hono docs page with:
- Main technology node (artifact/technology)
- Concepts (artifact/concept)  
- Code patterns (artifact/code_pattern)
- Constraints (knowledge/constraint)

**Current Flow**:
1. Call `texere_store_artifact` for technology + concepts
2. Save returned IDs
3. Call `texere_store_knowledge` for constraints
4. Manually reference saved IDs in edges

**Pain Point**: Can't create all nodes + cross-type edges atomically.

**Recommendation**: Add `texere_store_batch` that accepts heterogeneous node types:
```typescript
texere_store_batch({
  nodes: [
    { type: "artifact", role: "technology", temp_id: "hono", ... },
    { type: "knowledge", role: "constraint", temp_id: "zero_deps", ... }
  ],
  edges: [
    { source_id: "zero_deps", target_id: "hono", type: "ANCHORED_TO" }
  ]
})
```

### 2. No Bulk URL Fetching

**Scenario**: Need to fetch 6 Hono doc pages.

**Current Flow**:
1. Call webfetch for page 1, wait
2. Call webfetch for page 2, wait
3. ... repeat 6 times

**Pain Point**: Sequential fetching is slow. Could parallelize but requires orchestration.

**Observation**: Not Texere-specific, but relevant to ingestion workflow.

**Recommendation**: (Out of scope for Texere) Could use background tasks or batch fetch helper.

## Conclusion

Texere's knowledge graph ingestion workflow is **functional but has rough edges**. The core workflow (search → create nodes → create edges) works, but multi-type ingestion, temp_id scoping, and lack of automation create friction.

**Biggest Win**: Atomic node+edge creation with temp_ids (within single call) is excellent.

**Biggest Pain**: temp_id scope limitation breaks atomic workflow for cross-type relationships.

**Overall Grade**: **B+** - Solid foundation, clear room for improvement in DX polish.
